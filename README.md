# Доработки визуала и ограничения по изображениям

В рамках текущей задачи **не создавались и не изменялись бинарные файлы**.
Использованы только уже существующие изображения из репозитория по исходным путям.

## Что уже подключено
- Элементы для верхнего фона HERO: `../Элементы на главный верхний фон HErobg/*.png`
- Элементы для фоновых секций SITEBG: `../Элементы на фон SITEBG Все что под нижним фоном/*.png`
- Иконки блоков «Программа / Локации / Отзывы»: `../Элементы в блоки/*.png`
- Фото тренеров: `../Фото тренеры/*.png`

## Требуется добавить вручную
- `Фото Выпускники/` (набор изображений для раздела «Выпускники»)
  - описание: "фото выпускников для карточек траекторий"
  - формат: webp (рекомендуется) или png/jpg
  - куда вставить: `Site/assets/js/data.js` → массив `window.SITE_DATA.result.trajectories[].photo`

- `Фото/` (набор изображений для раздела «Фото и видео»)
  - описание: "фото для слайдера блока Фото и видео"
  - формат: webp (рекомендуется) или png/jpg
  - куда вставить: `Site/assets/data/gallery.json` → `photoSlides[].src`

- `фото Тестирование/` (набор изображений для раздела «Тестирование»)
  - описание: "карточки/фото для слайдера тестирования"
  - формат: webp (рекомендуется) или png/jpg
  - куда вставить: `Site/assets/data/gallery.json` → `testingSlides[].src`

## Рекомендации по производительности
- Для всех новых изображений использовать modern-формат `webp`.
- Целевой вес одного элемента: до ~150–250 KB (фоновые до ~300 KB, если без потери качества нельзя).
- Добавлять `loading="lazy"` для всех не-критичных изображений.
- Избегать избыточного количества декоративных элементов в одном viewport (особенно mobile).

## Предупреждение заказчику по нагрузке
Чем больше декоративных картинок на HERO и SITEBG, тем выше общий вес страницы и время первой отрисовки на мобильных сетях. Даже в `webp` большое число элементов может увеличить LCP/CLS и замедлить загрузку.
